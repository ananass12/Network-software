# Security Audit Report

## Объект аудита (что именно проверялось)

- **Проект**: сервис из `weeks/week-15/starter/`
  - **gRPC**: `service.py` (LikesService: `CreateLike`, `GetLike`, `SubscribeLikes`)
  - **REST**: `rest_app.py` (FastAPI: `POST /api/likes`)
- **Хранилище**: in-memory `dict` (`likes_storage`) без БД и без пользователей
- **Развёртывание**: `weeks/week-15/docker-compose.yml` публикует порты `8174` (gRPC) и `8080` (REST) наружу

## Чек-лист (OWASP Top 10, 10–15 проверок)

- [x] **A01 Broken Access Control**: есть ли авторизация на чтение/создание?  
  - **Результат**: **нет** (любой клиент может читать/создавать).
- [x] **A02 Cryptographic Failures**: используется ли TLS (HTTPS/gRPC-TLS), не передаются ли данные в plaintext?  
  - **Результат**: **нет TLS** (используется `add_insecure_port`, REST обычно также без HTTPS).
- [x] **A03 Injection**: есть ли SQL/Command/Template injection через пользовательский ввод?  
  - **Результат**: **SQL нет** (нет БД), **Command нет** (нет системных вызовов). Но **нет валидации** и есть риск DoS через большие payload.
- [x] **A04 Insecure Design**: есть ли модель угроз, лимиты, защитные меры в дизайне (rate-limit, quotas)?  
  - **Результат**: **нет** (сервис “доверяет” клиенту и бесконтрольно растит память).
- [x] **A05 Security Misconfiguration**: лишние открытые порты/наружная экспозиция, dev-режимы, дефолтные настройки?  
  - **Результат**: порты опубликованы наружу; нет ограничений по сети, нет auth, нет TLS.
- [x] **A06 Vulnerable and Outdated Components**: фиксируются ли версии зависимостей/есть ли скан?  
  - **Результат**: в `requirements.txt` стоят **диапазоны `>=`**, что ухудшает воспроизводимость и контроль уязвимостей.
- [x] **A07 Identification & Authentication Failures**: есть ли аутентификация (API key/JWT/mTLS), защита от подделки клиента?  
  - **Результат**: **нет**.
- [x] **A08 Software and Data Integrity Failures**: защита CI/CD, проверка целостности артефактов, подписи образов?  
  - **Результат**: **не настроено/не видно** по текущим файлам.
- [x] **A09 Security Logging and Monitoring Failures**: есть ли structured logs, аудит доступа, алерты?  
  - **Результат**: только `print(...)`, нет корреляции запросов, нет журналирования ошибок/клиентов.
- [x] **A10 SSRF**: есть ли исходящие запросы по URL от пользователя?  
  - **Результат**: **нет** (не применимо в текущем коде).
- [x] **gRPC/REST Input Validation**: проверка схемы, ограничения размера/длины строк, reject пустых/подозрительных значений  
  - **Результат**: **нет** (REST принимает `dict`, gRPC принимает строку без ограничений).
- [x] **DoS/Resource Exhaustion**: защита от переполнения памяти/длинных соединений/стримов  
  - **Результат**: **нет** (неограниченный `likes_storage`, стримы можно держать, нет лимитов).
- [x] **Transport/Network Hardening**: ограничения ingress, internal-only порты, firewall/compose сети  
  - **Результат**: **нет** (compose публикует сервисы, сегментации нет).

## Попытки “взломать себя” (что пробовала/что ожидалось)

- **Подделка токена/JWT**: невозможна как проверка, потому что **токенов нет вообще** - фактически любой клиент “авторизован”.
- **Перехват трафика (MitM)**: при работе по HTTP/не‑TLS gRPC данные **читаются в plaintext** на пути (особенно в публичной сети).
- **Инъекции**:
  - **SQL injection**: не применимо (нет SQL).
  - **Command injection**: не применимо (нет shell/exec).
  - **DoS через ввод**: отправка очень длинного `target`/массовые запросы ведут к росту RAM (и логов).
- **Broken Access Control**: `GetLike(id)` позволяет читать любой ID без прав; в REST можно создавать записи без ограничений.

## Найденные проблемы (уязвимость - риск - решение)

### 1) Нет аутентификации и авторизации (Broken Access Control / Auth failures)
- **Где**: `weeks/week-15/starter/service.py`, `weeks/week-15/starter/rest_app.py` — отсутствуют проверки личности клиента и прав.
- **Риск**: **Критический** для реального сервиса (любой может создавать/читать данные, спамить, ломать статистику).
- **Решение**:
  - **Минимум**: API key в заголовке + проверка на сервере.
  - **Лучше**: JWT (проверка подписи, срока, `aud/iss`), роли/права (RBAC).
  - Для gRPC в проде часто используют **mTLS** (клиентские сертификаты) или JWT через metadata/interceptors.

### 2) Нет шифрования транспорта (Cryptographic Failures / MitM)
- **Где**: gRPC использует `server.add_insecure_port(...)`, REST в compose обычно будет по HTTP.
- **Риск**: **Высокий** (перехват/подмена запросов, утечка данных/идентификаторов).
- **Решение**:
  - Включить **TLS**: HTTPS для REST, `add_secure_port` для gRPC.
  - В окружениях типа k8s — TLS терминатор (Ingress) + internal mTLS при необходимости.

### 3) Нет валидации входных данных и типизации (Insecure Design)
- **Где**: REST принимает `payload: dict` и берёт `payload.get("target", "")`; gRPC принимает `request.target` без ограничений.
- **Риск**: **Средний - Высокий** (ошибки логики, DoS большими payload, неожиданные данные).
- **Решение**:
  - REST: Pydantic-модель с ограничениями (min/max length, regex), возврат 422 на невалидный ввод.
  - gRPC: валидация на сервере (длина `target`, допустимые символы), возвращать `INVALID_ARGUMENT`.

### 4) Возможен DoS по памяти/ресурсам (Resource Exhaustion)
- **Где**: `likes_storage` растёт без ограничений, нет TTL/лимитов; стрим `SubscribeLikes` можно массово открывать.
- **Риск**: **Высокий** (вывод сервиса из строя).
- **Решение**:
  - Лимит размера хранилища/TTL/очистка; квоты на клиента.
  - Rate limit (на уровне API gateway / reverse proxy).
  - Ограничение размеров сообщений (FastAPI/uvicorn, gRPC max message size), таймауты.

### 5) Недостаточное логирование и мониторинг (Logging & Monitoring)
- **Где**: используется `print`, нет IP/identity/request-id, нет уровней логов, нет ошибок в формате.
- **Риск**: **Средний** (сложно расследовать инциденты/атаки, заметить аномалии).
- **Решение**:
  - Structured logging (json), request-id, уровни (info/warn/error).
  - Метрики (RPS, latency, error-rate), алерты на всплески.

### 6) Риск “плавающих” зависимостей (Vulnerable/Outdated Components)
- **Где**: `weeks/week-15/starter/requirements.txt` использует `>=`.
- **Риск**: **Средний** (непредсказуемые обновления, сложнее контролировать CVE и воспроизводимость).
- **Решение**:
  - Зафиксировать версии (pin), добавить SCA (pip-audit/Dependabot) и регулярное обновление по расписанию.

### 7) Экспозиция сервисов наружу без ограничений (Security Misconfiguration)
- **Где**: `weeks/week-15/docker-compose.yml` публикует `8174:8174` и `8080:8080`.
- **Риск**: **Высокий** (увеличение поверхности атаки).
- **Решение**:
  - Ограничить публикацию (только localhost или internal network), использовать gateway/reverse-proxy с TLS+auth.

## Итог

Сервис функционально работает, но **в текущем виде небезопасен для реального окружения**: отсутствуют аутентификация/авторизация и TLS, нет валидации и лимитов, возможен DoS, слабое логирование. Рекомендуемый минимум для “боевого” состояния: **TLS + auth (JWT/API key) + валидация входа + rate-limit/лимиты + нормальные логи**.
