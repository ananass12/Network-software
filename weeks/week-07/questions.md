# Вопросы для самопроверки

1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?

Файл .proto — это единый контракт (интерфейс), независимый от языка программирования.
Основные причины:

- Кросс-языковость. Один .proto файл может сгенерировать код для Python, Go, Java, C++ и т.д. Все сервисы будут понимать друг друга.
- Единый источник правды. Контракт описывается один раз. Нельзя случайно изменить тип поля только на клиенте или только на сервере.
- Автоматическая сериализация. Компилятор сам генерирует эффективный код для преобразования объектов в байты и обратно.
- Валидация на этапе компиляции. Если контракт изменился, код не скомпилируется, пока вы не перегенерируете его.

2. Что такое Unary RPC? Чем он концептуально похож на REST запрос?

Unary RPC — это обычный вызов удаленной процедуры: один запрос → один ответ.

Сходства с REST:
- Паттерн: Запрос - Ответ.
- Блокирование: Клиент ждёт ответа.
- Использование: CRUD операции.

Различия:

- REST работает с ресурсами (HTTP-глаголы: GET /likes, POST /likes).
- gRPC работает с методами (вызов функций: CreateLike(), GetLike()).

3. Почему gRPC обычно эффективнее REST (JSON) по процессору и трафику?

Сравнение характеристик:
- Формат данных. REST использует текстовый JSON (читаемый человеком). gRPC использует бинарный Protobuf (компактный).
- Размер сообщения. В JSON имена полей повторяются в каждом сообщении. В Protobuf поля кодируются номерами, что занимает меньше места.
- Парсинг. Текст в объект (JSON) парсится медленнее. Байты в объект (Protobuf) парсятся быстрее.
- Схема данных. В REST схема опциональна (Swagger). В gRPC схема обязательна (.proto).ы

4. Что такое "обратная совместимость" в Protobuf и зачем нужны номера полей (тэги, например `= 1`)?

Номера полей (тэги) — это идентификаторы, которые используются в бинарном формате для связи данных.

Правила работы с полями:
- Не менять номера. Если изменить = 1 на = 2, старые клиенты не смогут прочитать данные.
- Не удалять поля. Удалённое поле может понадобиться позже. Лучше пометить как reserved.
- Можно добавлять. Новые поля с новыми номерами безопасны — старые клиенты их просто игнорируют.

Сценарии обратной совместимости:
- Старый клиент + Новый сервер = Работает (клиент игнорирует новые поля).
- Новый клиент + Старый сервер = Работает (сервер игнорирует неизвестные поля).

5. Что такое Stub (стаб) на клиенте и Servicer на сервере?

Stub (Клиент) - прокси-объект.

Выглядит как локальный объект, но вызывает удаленный метод.

Servicer (Сервер) - базовый класс.

От него вы наследуетесь и реализуете бизнес-логику.

6. В каких случаях вы бы выбрали REST, а в каких — gRPC?

REST, если:

- публичный API для внешних клиентов.
- нужно поддерживать мобильные приложения или браузеры напрямую.
- важна человекочитаемость (JSON легко читать и отлаживать).
- сервис простой и не требует высокой производительности.

gRPC, если:

- внутренняя коммуникацию микросервисов.
- требуется высокая производительность и низкая задержка.
- нужна строгая типизация контрактов.
- требуется стриминг данных (двусторонний поток).
- сервисы написаны на разных языках программирования.

Кратко:
- REST — для публичных API, веб-фронтенда, простых сервисов.
- gRPC — для внутренней коммуникации микросервисов, высоконагруженных систем.
