# Вопросы для самопроверки

1. Почему мы создаем Deployment, а не сразу Pod?

Pod — это минимальная единица развертывания в K8s, но он «эфемерен» (временен). Если под упадет или нода перезагрузится, сам по себе он не восстановится.

Deployment — это контроллер более высокого уровня, который:
Управляет жизненным циклом: Гарантирует, что указанное количество реплик (подов) всегда запущено. Если под падает, Deployment создает новый.
Обеспечивает обновления: Позволяет безопасно обновлять версию приложения (Rolling Update) без простоя.
Масштабируемость: Позволяет легко менять количество копий приложения (replicas).

2. В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?

Liveness Probe - Отвечает на вопрос «Живо ли приложение?».

Если провалилась: Kubernetes считает, что приложение зависло или работает некорректно, и перезагружает (restarts) контейнер.

Readiness Probe - Отвечает на вопрос «Готово ли приложение принимать трафик?».

Если провалилась: Kubernetes не перезагружает под. Он просто исключает этот под из списка эндпоинтов Service. На этот под перестают приходить запросы от других сервисов, пока он снова не станет готовым. Это критично во время старта приложения или временных пиков нагрузки.

3. Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?

Requests (Запросы): Гарантированный минимум ресурсов, который K8s резервирует для пода при планировании (scheduling). Планировщик ищет ноду, где есть хотя бы столько свободных ресурсов.

Limits (Лимиты): Жесткий потолок потребления ресурсов.

4. Как Service находит свои Pod-ы? (Подсказка: Labels & Selectors).

Service использует механизм Label Selector (селектор меток).

- В deployment.yaml у подов есть метки (labels), например: app: profiles-app.
- В service.yaml в поле spec.selector указывается та же пара ключ-значение: app: profiles-app.
- Kubernetes постоянно отслеживает поды с такими метками и автоматически добавляет их IP-адреса в эндпоинты сервиса. Если под удаляется или создается новый, сервис обновляет список автоматически.

5. Какой тип сервиса (ClusterIP, NodePort, LoadBalancer) используется для внутреннего общения, а какой — для внешнего доступа?

- ClusterIP (по умолчанию): Выделяет виртуальный IP только внутри кластера. Используется для внутреннего общения между микросервисами (например, Gateway обращается к Profiles Service). Внешний доступ к нему невозможен.

- NodePort: Открывает порт на каждой ноде кластера. Позволяет получить доступ извне по IP ноды. Часто используется для тестирования.

- LoadBalancer: Создает внешний балансировщик нагрузки (в облачных провайдерах) с внешним IP. Используется для внешнего доступа продакшн-сервисов.

6. Что такое "Rolling Update" в контексте Деплоймента?

Rolling Update (Поэтапное обновление) — это стратегия обновления приложения без остановки сервиса (zero-downtime).

При изменении образа контейнера в Deployment:

- K8s создает новый под с новой версией.

- Ждет, пока новый под пройдет Readiness Probe (станет готовым).

- Удаляет один старый под.

- Повторяет шаги 1–3, пока все поды не будут обновлены.
