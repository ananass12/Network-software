# Вопросы для самопроверки

1. В чем разница между Continuous Integration, Continuous Delivery и Continuous Deployment?

- Continuous Integration (Непрерывная интеграция): Частое слияние кода всех разработчиков в общую ветку (main). Каждый пуш автоматически проверяется: линтинг, сборка, запуск тестов. Цель: Найти ошибки как можно раньше.

- Continuous Delivery (Непрерывная доставка): Код всегда находится в состоянии, готовом к релизу. После успешного CI артефакт (Docker-образ) собирается и сохраняется. Развертывание на прод происходит вручную (по кнопке). Цель: Снизить риск релиза, сделав его рутинной процедурой.

- Continuous Deployment (Непрерывное развертывание): Полная автоматизация. Если тесты прошли, код автоматически выкатывается на продакшен без участия человека. Цель: Максимальная скорость доставки фич пользователю.

2. Почему тесты должны запускаться *до* сборки Docker образа (или хотя бы до его пуша)?

- Экономия времени и ресурсов: Сборка Docker-образа (особенно с установкой зависимостей и компиляцией) занимает минуты. Запуск pytest — секунды. Нет смысла ждать сборку тяжелого образа, если код падает на элементарном юнит-тесте.

- Принцип Fail Fast: Чем раньше мы узнаем об ошибке, тем дешевле её исправить.

- Чистота реестра: Если собирать образ до тестов, в Docker Registry (или в списке артефактов) будут появляться "битые" образы с нерабочим кодом, которые потом придется чистить вручную.

3. Зачем нужно кэширование зависимостей в CI пайплайне?

В CI/CD окружение каждый раз создается "с нуля" (чистый Ubuntu-контейнер).

При каждом коммите pip install будет заново скачивать сотни мегабайт пакетов из интернета. Это медленно (лимиты сети) и нестабильно (если PyPI упадет, пайплайн упадет).

4. Что такое "Secret" в контексте CI/CD? Как безопасно передать пароль от Docker Hub в пайплайн?

Secret - чувствительные данные (пароли, API-токены, ключи SSH), которые нельзя хранить в коде репозитория в открытом виде.

Как передать безопасно:
- В настройках репозитория переменная, например, DOCKER_PASSWORD.

- В YAML-файле пайплайна через специальный синтаксис: ${{ secrets.DOCKER_PASSWORD }}.


5. Что такое "Matrix Build" (матричная сборка)?

Matrix Build - возможность запустить один и тот же пайплайн параллельно с разными параметрами. Чтобы убедиться, что код работает везде.


6. Почему нельзя хранить конфиги продуктовой среды в репозитории кода (или можно, но осторожно)?

- В конфигах часто лежат пароли от баз данных, ключи шифрования, адреса внутренних сервисов. Если репозиторий утечет (или станет случайно публичным), злоумышленники получат доступ к проду.

- Код приложения (Image) должен быть одинаковым для Dev, Stage и Prod. Меняются только конфигурации (переменные окружения, конфиги БД).
