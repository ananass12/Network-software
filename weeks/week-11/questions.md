# Вопросы для самопроверки

1. Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?

- Можно создать несколько сетей (например, frontend и backend). Базу данных можно подключить только к backend, чтобы она была недоступна напрямую из внешних сервисов или шлюза, если это не требуется.

- В пользовательских сетях Docker встроенный DNS-сервер работает надежнее. Контейнеры видят друг друга по именам сервисов (например, http://db:5432). В дефолтной сети это тоже работает, но при подключении контейнеров из разных docker-compose.yml файлов или внешних контейнеров явные сети упрощают управление.

- Можно явно указать драйвер (обычно bridge) и его параметры (подсеть, шлюз), что важно для сложных инфраструктур, где нужно избегать конфликтов IP-адресов.

2. В чем разница между `build` и `image` в описании сервиса?

- build: Указывает Docker Compose собрать образ локально из исходного кода перед запуском.

- image: Указывает Compose скачать готовый образ из реестра (Docker Hub, GitLab Registry и т.д.).

3. Что делает `depends_on`? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?

depends_on управляет порядком запуска контейнеров. Docker Compose запустит зависимые сервисы (например, БД) перед тем, как запустить основной сервис (приложение).

Он гарантирует только старт контейнера, а не готовность приложения внутри него.

4. Как работают healthcheck-и и как использовать их в `depends_on`?

Healthcheck - команда, которая периодически выполняется внутри контейнера для проверки его работоспособности 

Статусы: starting - healthy / unhealthy.

5. Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?

- Явно в файле docker-compose.yml (секция environment)

- Через файл .env

6. Почему Docker Compose редко используют в продакшене (хотя это возможно)?

Docker Compose предназначен для разработки и простых стендов (staging). В высоконагруженном продакшене используют оркестраторы вроде Kubernetes (K8s).

Причины:

- Compose не умеет автоматически перезапускать упавшие сервисы на других нодах (серверах), если машина вышла из строя. K8s умеет.

- В Compose сложно динамически масштабировать сервисы (запускать 10 копий API вместо 1) без ручного вмешательства и настройки балансировщиков. K8s делает это автоматически (HPA).

- Compose не поддерживает "синие-зеленые" деплои или канареечные релизы из коробки. При обновлении (up --build) сервис перезапускается с простоем.

- В K8s есть мощные механизмы для безопасного хранения секретов и конфигов, которые лучше интегрируются с облачными провайдерами.
