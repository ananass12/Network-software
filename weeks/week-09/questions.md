# Вопросы для самопроверки

1. Зачем нужен Signaling сервер, если WebRTC — это P2P технология?

Signaling‑сервер нужен только для установки соединения, а не для передачи медиаданных.
Через него два браузера обмениваются:

SDP‑описаниями (offer/answer),

ICE‑кандидатами (IP + порты, способы соединения).

После этого WebRTC‑соединение устанавливается напрямую между пирами P2P, а сервер больше не участвует в трафике.

2. Что такое NAT и почему он мешает прямому соединению?

NAT (Network Address Translation) — это механизм, при котором многие устройства в локальной сети используют один внешний IP‑адрес для выхода в интернет.

Проблема для P2P:

- внутренние IP‑адреса «невидимы» извне

- входящие соединения просто не знают, как их пробросить

Поэтому два пира «за NAT» не могут напрямую подключиться друг к другу, если не применять специальные протоколы (STUN / TURN / ICE).

3. В чем разница между STUN и TURN серверами?

- STUN определяет публичный IP и порт клиента за NAT, формирует кандидат и помогает установить прямое P2P‑соединение. Если это работает, весь трафик идёт напрямую между пирами.

- TURN используется, когда STUN не срабатывает (например, симметричный NAT): TURN‑сервер ретранслирует трафик между клиентами, выступая как прокси. Это гарантирует связность, но создаёт нагрузку на сервер и увеличивает задержки.

На практике ICE‑клиент сначала пробует STUN, а TURN — только как резервный вариант.

4. Что такое ICE Candidate?

ICE‑кандидат — это один из возможных адресов (IP + порт + транспорт), по которому один пир может быть доступен другому.
Каждый участник WebRTC‑сессии генерирует несколько кандидатов:

- host (локальный IP + порт),

- server‑reflexive (через STUN),

- relay (через TURN).

ICE‑механизм перебирает пары кандидатов и выбирает наилучший путь для прямого соединения между пирами.

5. Почему WebRTC обычно использует UDP, а не TCP?

UDP даёт меньшую задержку и не вводит «переупорядочивания и переотправки» пакетов, что критично для аудио/видео в реальном времени (если один кадр пришёл с опозданием — его проще пропустить, чем ждать по TCP‑правилам).

WebRTC реализует собственную логику доставки и надёжности поверх UDP, поэтому ему не нужна гарантия «как в TCP».

TCP чаще используется в WebRTC только как резервный транспорт, если UDP заблокирован сетью.

6. Что такое SDP (Session Description Protocol) и какая информация в нем содержится?

SDP (Session Description Protocol) — это текстовый формат описания мультимедийной сессии (какую аудио/видео кодеки, транспорт, форматы, ориентиры по offer/answer) WebRTC‑сигнализации.

В offer и answer‑SDP обычно содержится:

- список медиапотоков (аудио/видео),

- поддерживаемые кодеки и их параметры,

- тип сессии (sendonly, sendrecv),

- информация о транспорте (UDP, порты, ICE‑кандидаты).

Через Signaling‑сервер стороны обмениваются этими SDP‑описаниями, чтобы согласовать параметры до начала передачи реального медиатрафика.
