# Анализ производительности

Вариант: `photos-s18`  
Фактически измеряемые сервисы в текущем стенде: REST и gRPC для лайков из `week-15/starter`.

## Цель

Сравнить `latency` и `throughput` для двух реализаций:

- REST: `POST /api/likes` на `http://127.0.0.1:8080`
- gRPC: `LikesService/CreateLike` на `127.0.0.1:8174`

Тесты запускаются при разной конкуренции: `1`, `10`, `100`, `200`, `500`.

## Условия теста

- Docker Compose из `weeks/week-15/docker-compose.yml`
- REST-инструмент: `wrk`
- gRPC-инструмент: `ghz`
- Длительность одного прогона: `15s` или `30s`
- Payload REST: `{"target":"post_123"}`
- Payload gRPC: `{"target":"post_123"}`

## Команды запуска

Поднять сервисы:

```bash
cd "/home/nastya/Network software/weeks/week-15"
docker compose up -d --build
```

Запустить REST-тесты:

```bash
cd "/home/nastya/Network software/weeks/week-15/start"
chmod +x test_rest.sh test_grpc.sh
./test_rest.sh
```

Запустить gRPC-тесты:

```bash
cd "/home/nastya/Network software/weeks/week-15/start"
./test_grpc.sh
```

Вывод каждого прогона сохраняется в `weeks/week-15/start/results/` (файлы вида `rest_c{N}.txt` и `grpc_c{N}.txt`).

Если хотите прогнать только один уровень конкуренции:

```bash
./test_rest.sh 10
./test_grpc.sh 10
```

## Анализ роста latency

- При увеличении concurrency `latency` обычно растет медленно до некоторого предела.
- После точки насыщения RPS почти перестает расти, а `latency` начинает увеличиваться резко.
- Для итогового вывода нужно сравнить, на каком уровне нагрузки это произошло у REST и у gRPC.

Пример интерпретации:

1. Пока `Requests/sec` растет почти пропорционально, сервер еще не насыщен.
2. Когда рост RPS замедляется, а `P95/P99 latency` резко увеличиваются, это и есть зона насыщения.
3. Если у gRPC при той же concurrency выше RPS и ниже P95/P99, значит для этого сценария он эффективнее REST.

## Точка насыщения

- REST: c10-c100 RPS +14% (3776→4289), P99 x14 (4ms-55ms), дальше RPS почти плато, но хвосты/таймауты растут (на `c=200` уже `P99=732ms`, есть `timeout=28`)

- gRPC: c10-c100 RPS -8% (2993→2762), но `P99` растет x15 (4ms-67ms), а дальше `P99` и ошибки растут (на `c=500` `P99≈201ms`, много `Unavailable`)

Дополнительные прогоны на `c=200` и `c=500` подтверждают:

- сохранится ли плато по RPS (упрется ли CPU/IO/очереди),
- насколько «утяжелятся хвосты» `P95/P99`,
- появятся ли ошибки (например, `Unavailable` у gRPC) и при какой нагрузке.


Фактически по этим замерам (15s, localhost):

1. **Throughput**: REST выше на всех уровнях конкуренции (например, `c=10`: 3776 RPS vs 2993 RPS; `c=100`: 4289 RPS vs 2762 RPS). При росте concurrency после `c≈100` RPS у обеих реализаций почти не растет (плато).
2. **Latency**:
   - на `c=1` обе реализации быстрые (sub-ms..few ms), REST быстрее по RPS;
   - на `c=10` обе держат низкий `P99` (REST ~4ms, gRPC ~4.5ms);
   - начиная с `c=100` заметна деградация хвостов: у REST `P99` ~55ms, у gRPC `P99` ~67ms;
   - на `c=200`/`c=500` рост latency продолжает ускоряться, появляются ошибки: у REST таймауты (`timeout=28` на `c=200`, `timeout=95` на `c=500`), у gRPC растет число `Unavailable` (188 на `c=200`, 474 на `c=500`).
3. **Насыщение**: около `c≈100` для обоих сервисов — RPS выходит на плато, а `P95/P99` начинают расти значительно быстрее.
