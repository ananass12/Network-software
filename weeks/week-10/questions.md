# Вопросы для самопроверки

1. Что такое слой в Docker и почему порядок команд в Dockerfile важен для кэширования?

Слой (Layer) — это неизменяемый пакет файловых изменений, создаваемый каждой инструкцией (RUN, COPY, ADD) в Dockerfile. Образ состоит из стека таких слоев, накладывающихся друг на друга.

Docker кэширует результаты выполнения инструкций. Если инструкция и её контекст (файлы) не изменились с предыдущей сборки, Docker берет результат из кэша, а не выполняет команду заново.

2. В чем суть Multi-stage build? Какие проблемы он решает?

Использование нескольких инструкций FROM в одном Dockerfile. Каждый этап (stage) создает промежуточный образ. Из финального образа копируются только нужные артефакты (бинарники, установленные библиотеки, скомпилированный код) из предыдущих этапов.

Решаемые проблемы:
- В финальный образ не попадают инструменты сборки (компиляторы, линтеры, заголовочные файлы), которые нужны только на этапе компиляции/установки.
- В продакшн-образе нет исходников тестов, конфигов разработки и инструментов отладки.
- Позволяет иметь один Dockerfile вместо двух (один для сборки, другой для запуска).

3. Зачем нужен файл `.dockerignore`?

Файл .dockerignore исключает указанные файлы и папки из контекста сборки.

4. Чем отличаются инструкции `CMD` и `ENTRYPOINT`?

ENTRYPOINT: Задает неизменяемую исполняемую программу. Аргументы, переданные при запуске docker run, добавляются после команды entrypoint.

CMD: Задает аргументы по умолчанию для ENTRYPOINT или команду по умолчанию, если ENTRYPOINT не задан. Аргументы при запуске docker run полностью заменяют содержимое CMD.

5. Почему рекомендуется использовать `COPY` вместо `ADD`?

Инструкция ADD имеет дополнительный скрытый функционал, который часто не нужен.

Почему COPY лучше:
- COPY делает только одно — копирует файлы из контекста в образ. Это делает Dockerfile более читаемым и предсказуемым.
- Официальная документация советует использовать COPY, если вам не нужна автоматическая распаковка tar-архивов.

6. Почему приложение в контейнере должно слушать `0.0.0.0`, а не `127.0.0.1`?

- 127.0.0.1 (localhost): Означает "только внутри текущего сетевого пространства имен". Если приложение слушает 127.0.0.1 внутри контейнера, оно будет доступно только процессам внутри этого же контейнера. Хост-машина и другие контейнеры не смогут к нему подключиться, даже с пробросом портов (-p).
- 0.0.0.0: Означает "слушать на всех доступных сетевых интерфейсах". Внутри контейнера это позволяет принимать соединения, приходящие извне (от хост-машины через проброшенный порт).
