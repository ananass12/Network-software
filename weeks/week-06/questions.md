# Вопросы для самопроверки

1. Зачем нужны `variables` в GraphQL запросе? Почему нельзя просто склеивать строки?

Переменные (Variables) нужны для того, чтобы сделать запросы динамическими и безопасными, отделяя логику запроса от пользовательских данных.

- Если вы просто склеиваете строки, злоумышленник может попытаться внедрить вредоносный код в ваш запрос (аналог SQL-инъекций). При использовании переменных GraphQL гарантирует, что данные будут интерпретироваться строго как значения, а не как часть синтаксиса запроса. Параметры проходят через механизм валидации и экранирования.

- Переменные строго типизированы. Вы объявляете $id: ID!, и сервер проверит, соответствует ли переданное значение типу ID и не равно ли оно null.

- Если отделить переменные от тела запроса, клиент может сохранить строку запроса (query string) как постоянный идентификатор. 

2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?

Тело POST-запроса обычно представляет собой JSON-объект со следующими полями:

- query (строка, обязательно) - сам текст GraphQL-запроса или мутации.

- variables (объект, опционально) - значения для переменных, используемых в запросе.

3. Почему у GraphQL обычно только один URL endpoint?

В REST подходе ресурсы идентифицируются через разные URL (например, /users, /users/1/posts). GraphQL же использует единую точку входа (Single Endpoint) по следующим причинам:

- GraphQL API работает не с ресурсами, а с графом данных. У вас есть одна "точка входа" в этот граф, а дальнейшая навигация происходит по ребрам графа (полям), указанным в запросе.

- Создание, обновление и удаление данных (которые в REST были бы разными эндпоинтами) в GraphQL — это просто поля в корневом типе Mutation. Логика разделения операций идет не через URL, а через типы операций (query vs mutation).

4. Что означает наличие поля `errors` в ответе сервера? Может ли оно прийти вместе с `data`?

Поле errors означает, что при выполнении запроса возникли проблемы.Оно может прийти вместе с data.

В GraphQL возможна ситуация частичного успеха:

- Если ошибка произошла на этапе валидации запроса (синтаксис, типы), то data будет null, а в errors будет описание проблемы.

- Если ошибка произошла во время выполнения (например, упал один из резолюверов внутри большого запроса), то сервер вернет успешные данные в data (неудачные поля будут null), а описание ошибки — в массиве errors.

5. Как клиент должен обрабатывать ошибки, если HTTP статус ответа 200?

Алгоритм обработки:

- Получить ответ.

- Проверить, что статус HTTP успешен (200-299).

- Распарсить JSON.

- Обязательно проверить наличие массива errors в теле ответа.

- Если errors есть — обработать их (показать уведомление пользователю, залогировать, попробовать повторить запрос), даже если в data что-то пришло.

6. В чем плюсы использования типизированных клиентов (например, Apollo Client или Relay) перед простыми HTTP-запросами?

- Клиенты (особенно Apollo и Relay) автоматически кэшируют полученные объекты по их типу и ID. Если вы запросите пользователя в одном месте, а потом в другом, клиент может отдать данные из кэша или сходить только за новыми полями, не перезапрашивая всё.

- Хуки (useQuery, useMutation) предоставляют простые флаги (loading, error, data), избавляя от необходимости писать эту логику вручную.

- Клиенты упрощают логику обновления данных после изменений. Вы можете автоматически обновить кэш или перезапросить списки, не перезагружая страницу.

- Можно мгновенно показать пользователю результат его действия (до ответа от сервера), а потом "поверх" наложить реальный ответ. Это создает эффект мгновенной работы приложения.

- Клиенты могут объединять несколько запросов, выполненных в рамках одного "тика" event loop'а, в один HTTP запрос к серверу. А также отменять дублирующиеся запросы, идущие параллельно.
