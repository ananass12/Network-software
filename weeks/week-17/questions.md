# Вопросы для самопроверки

1. Какие критерии вы использовали при разделении монолита на микросервисы?

   - Выделила **границы по бизнес-ответственности**: внешний вход (`gateway`), домен/данные (`events-svc-s18`), внутренний контракт валидации (`events-grpc`).
   - Разделила по **разным SLA/нагрузке**: gateway масштабируется отдельно от БД-сервиса; gRPC можно масштабировать отдельно.
   - Минимизировала **связность**: gateway не знает про БД, а только про HTTP API; `events-svc` не зависит от реализации валидации, только от gRPC-контракта.

2. Почему вы выбрали именно этот стек технологий (язык, БД, протоколы)?

   - **Python + FastAPI**: быстро собрать REST и healthchecks, удобно документировать API.
   - **PostgreSQL**: простая и надёжная реляционная БД, логично для хранения сущностей `event`.
   - **REST**: удобен для внешних клиентов и тестирования.
   - **gRPC**: удобен для внутреннего взаимодействия сервисов с контрактом и строгими типами (валидация `location`).
   - **Docker Compose**: обеспечивает запуск “одной командой” локально.

3. Как ваши сервисы справляются с ошибками (retry, circuit breaker, graceful degradation)?

   - Сейчас реализовано базово:
     - Валидация `location` через gRPC: если gRPC вернул `ok=false` - `events-svc-s18` отвечает `400`.
     - Таймаут на gRPC вызов (`timeout=2`), чтобы не зависать.
   - Что можно добавить (как улучшение):
     - **Retry с backoff** на сетевые ошибки.
     - **Circuit breaker** при частых фейлах `events-grpc`.
     - **Graceful degradation**: принимать событие и отправлять `location` на асинхронную валидацию через очередь.

4. Как организован деплой и обновление вашей системы без простоя (Zero Downtime)?

   - Идея для K8s:
     - `Deployment` с `RollingUpdate` для stateless сервисов (`gateway`, `events-grpc`).
     - Для `events-svc-s18` тоже `RollingUpdate`, при условии корректных readiness-проб и совместимых изменений схемы.
     - Readiness/Liveness через `/health`.
     - Миграции — отдельным шагом (job) перед выкатыванием новой версии.

5. Что было самым сложным в интеграции всех компонентов вместе?

   - Согласовать контракты (REST - GraphQL - gRPC) и сделать так, чтобы Compose поднимал сервисы в правильном порядке (healthchecks/depends_on).
   - Настроить окружение так, чтобы gateway ходил по DNS-имени сервиса внутри сети Compose, а наружу публиковался только один порт.

6. Что бы вы улучшили, если бы у вас был еще месяц?

   - Добавила миграции (Alembic), полноценные тесты (pytest), метрики (Prometheus) и трассировку (OpenTelemetry).
   - Сделала асинхронную обработку через очередь (RabbitMQ/Kafka) для слабосвязанных задач.
   - Добавила авторизацию (JWT) и rate limiting на gateway.
